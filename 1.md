### 一、技术选型

**问题分析**
核心痛点在于 AI 缺乏对本地开发环境“可视”与“可控”的能力。传统方案中 AI 仅能生成代码，无法感知代码运行状态（端口占用、进程存活），也无法进行交互式干预（热重载输入 'r'、发送中断信号 Ctrl+C）。

**备选方案**

1. **Electron + node-pty + Express**：生态成熟，但体积臃肿，资源占用高。
2. **Tauri + SSH Client**：过度设计，增加网络层复杂度，不适合本地进程管理。
3. **Tauri + portable-pty + Axum**：Rust 原生高性能，体积小，且 Tauri 天然支持本地系统 API 调用，Axum 可在同一端口复用 MCP 服务与前端通信。

**最终选择 + 理由**
**Tauri + portable-pty + Axum (Rust)**

* **轻量级**：Tauri 打包体积远小于 Electron，适合作为常驻后台的“代理服务”。
* **系统级能力**：`portable-pty` 提供跨平台的伪终端实现，能直接接管进程的 STDIN/STDOUT/STDERR，这是实现 `Ctrl+C` 和输入 'r' 的基础。
* **协议原生支持**：Rust 的 `tokio` + `axum` 天然适合构建高并发的 MCP Server，处理 JSON-RPC 请求。

---

### 二、整体架构

架构核心是将 **GUI 终端** 与 **MCP 服务** 解耦，通过 **PTY Master** 这一单例进行桥接。

```mermaid
graph TD
    AI[AI Agent (MCP Client)] -->|JSON-RPC/HTTP| McpServer[MCP Server (Axum)]
    UI[Tauri WebView (Vue + xterm.js)] -->|Tauri Invoke| PtyService[PTY Service Singleton]
    McpServer -->|Read/Write| PtyService
    PtyService -->|Spawn| Shell[System Shell / User Process]
    Shell -->|Output Stream| PtyService
    PtyService -->|Broadcast| UI
    PtyService -->|Broadcast| McpServer
```

### 三、核心代码片段

#### [模块A: PTY 核心单例服务]

这是系统的核心，负责维护终端状态、处理输入输出流转。使用 `portable-pty` 创建原生 shell，并使用 `tokio::sync::broadcast` 实现输出的一对多分发（同时给 UI 和 MCP 消费）。

```rust
// src-tauri/src/pty_service.rs
use portable_pty::{native_pty_system, CommandBuilder, PtyPair, PtySize};
use std::sync::{Arc, Mutex};
use std::io::{Read, Write};
use tokio::sync::broadcast;
use anyhow::Result;

// 定义终端输出消息类型
#[derive(Clone, Debug)]
pub enum PtyOutput {
    Data(String),
    Exit,
}

pub struct PtyInstance {
    pub writer: Box<dyn Write + Send>,
    pub output_rx: broadcast::Receiver<PtyOutput>,
}

pub struct PtyService {
    // 使用 Arc<Mutex> 确保多线程安全访问 PTY 实例
    instance: Arc<Mutex<Option<PtyInstance>>>,
    // 广播通道，用于分发 PTY 输出
    output_tx: broadcast::Sender<PtyOutput>,
}

impl PtyService {
    pub fn new() -> Self {
        let (tx, _) = broadcast::channel(1024);
        Self {
            instance: Arc::new(Mutex::new(None)),
            output_tx: tx,
        }
    }

    /// 启动终端
    pub fn spawn(&self, cols: u16, rows: u16) -> Result<()> {
        let pty_system = native_pty_system();
        // 创建 PTY 对
        let pair = pty_system.openpty(PtySize { rows, cols, pixel_width: 0, pixel_height: 0 })?;
      
        let mut cmd = CommandBuilder::new_default_prog(); // 默认 shell
        let _child = pair.slave.spawn_command(cmd)?;
      
        // 获取读写句柄
        let mut reader = pair.master.try_clone_reader()?;
        let writer = pair.master.take_writer()?;
      
        // 保存实例
        let mut instance_lock = self.instance.lock().unwrap();
        *instance_lock = Some(PtyInstance {
            writer,
            output_rx: self.output_tx.subscribe(),
        });

        // 异步读取 PTY 输出并广播
        let tx = self.output_tx.clone();
        tokio::spawn(async move {
            let mut buf = [0u8; 4096];
            loop {
                match reader.read(&mut buf) {
                    Ok(0) => { let _ = tx.send(PtyOutput::Exit); break; },
                    Ok(n) => {
                        let data = String::from_utf8_lossy(&buf[..n]).to_string();
                        let _ = tx.send(PtyOutput::Data(data));
                    },
                    Err(e) => { eprintln!("Read error: {}", e); break; }
                }
            }
        });
      
        Ok(())
    }

    /// 写入数据 (支持字符、命令、特殊符号)
    pub fn write(&self, input: &str) -> Result<()> {
        let lock = self.instance.lock().unwrap();
        if let Some(inst) = lock.as_ref() {
            let mut writer = &inst.writer;
            writer.write_all(input.as_bytes())?;
            writer.flush()?; // 必须刷新，确保立即发送
        }
        Ok(())
    }
  
    /// 发送控制信号 (如 Ctrl+C)
    pub fn send_signal(&self, signal: &str) -> Result<()> {
        match signal {
            "SIGINT" => self.write("\x03"), // Ctrl+C
            "SIGQUIT" => self.write("\x1C"), // Ctrl+\
            _ => Ok(())
        }
    }
}
```

**设计理由**：

* **单例模式**：确保 AI 和 UI 操作的是同一个终端会话，避免状态不一致。
* **Broadcast Channel**：解决了“生产者（PTY）消费者（UI + MCP）”模型，避免数据竞争。
* **Signal 映射**：将抽象的信号（SIGINT）映射到底层字符编码（`\x03`），解决了用户提到的无法发送 Ctrl+C 的问题。

#### [模块B: MCP Server 实现 (Axum)]

实现标准的 MCP 协议，让 AI 能够调用终端能力。这里使用了 Axum 构建 HTTP 服务，并实现了 SSE (Server-Sent Events) 供 AI 长连接监听输出。

```rust
// src-tauri/src/mcp_server.rs
use axum::{
    routing::{get, post},
    Json, Router, extract::State,
    response::sse::{Event, Sse},
};
use serde::{Deserialize, Serialize};
use std::sync::Arc;
use tokio_stream::StreamExt;
use super::pty_service::{PtyService, PtyOutput};

// MCP 标准请求结构
#[derive(Deserialize)]
struct McpRequest {
    jsonrpc: String,
    method: String,
    params: Option<serde_json::Value>,
}

// MCP 工具调用参数
#[derive(Deserialize)]
struct ToolParams {
    name: String,
    arguments: serde_json::Value,
}

pub fn create_router(pty: Arc<PtyService>) -> Router {
    Router::new()
        .route("/sse", get(sse_handler))
        .route("/mcp", post(mcp_handler))
        .with_state(pty)
}

// 处理 AI 发起的工具调用
async fn mcp_handler(
    State(pty): State<Arc<PtyService>>,
    Json(req): Json<McpRequest>,
) -> Json<serde_json::Value> {
    // 简化的路由逻辑，实际项目中需根据 method 分发
    if req.method == "tools/call" {
        if let Some(params) = req.params {
            let args: ToolParams = serde_json::from_value(params).unwrap();
          
            // 根据工具名称执行操作
            match args.name.as_str() {
                "terminal_write" => {
                    let input = args.arguments["input"].as_str().unwrap_or("");
                    let _ = pty.write(input);
                    return Json(serde_json::json!({"status": "written"}));
                },
                "terminal_signal" => {
                    let sig = args.arguments["signal"].as_str().unwrap_or("");
                    let _ = pty.send_signal(sig);
                    return Json(serde_json::json!({"status": "signal_sent"}));
                },
                _ => {}
            }
        }
    }
    Json(serde_json::json!({"error": "unknown method"}))
}

// 供 AI 订阅终端输出流
async fn sse_handler(
    State(pty): State<Arc<PtyService>>,
) -> Sse<impl tokio_stream::Stream<Item = Result<Event, std::convert::Infallible>>> {
    let rx = pty.output_tx.subscribe();
  
    let stream = tokio_stream::wrappers::BroadcastStream::new(rx).filter_map(|msg| {
        match msg.ok()? {
            PtyOutput::Data(d) => Some(Ok(Event::default().data(d))),
            _ => None
        }
    });
  
    Sse::new(stream)
}
```

**设计理由**：

* **标准 JSON-RPC**：完全兼容 MCP 协议规范，确保主流 AI Agent 可直接接入。
* **SSE 流式输出**：AI 无需轮询，通过长连接实时获取终端的 STDOUT，解决了“难以理解界面”的问题，AI 可以“看到”命令执行的结果。

#### [模块C: 前端 UI 集成]

前端仅仅是一个展示层，通过 Tauri Command 与后端通信。

```javascript
// src/App.vue
<script setup>
import { onMounted, ref } from 'vue';
import { Terminal } from 'xterm';
import { FitAddon } from 'xterm-addon-fit';
import { invoke } from '@tauri-apps/api/tauri';
import { listen } from '@tauri-apps/api/event';

const terminalContainer = ref(null);

onMounted(async () => {
  const term = new Terminal({ cursorBlink: true });
  const fitAddon = new FitAddon();
  term.loadAddon(fitAddon);
  term.open(terminalContainer.value);
  fitAddon.fit();

  // 1. 初始化后端 PTY
  await invoke('init_pty', { cols: term.cols, rows: term.rows });

  // 2. 监听后端推送到前端的数据
  await listen('pty-output', (event) => {
    term.write(event.payload);
  });

  // 3. 前端输入转发给后端
  term.onData(async (data) => {
    await invoke('write_pty', { input: data });
  });
});
</script>

<template>
  <div ref="terminalContainer" class="terminal-container"></div>
</template>

<style>
.terminal-container { width: 100%; height: 100vh; }
</style>
```

**设计理由**：

* xterm.js 是前端终端的标准库，渲染性能极高。
* 逻辑极简：只做数据的双向转发（UI <-> Rust），不包含任何业务逻辑，方便后续扩展。
